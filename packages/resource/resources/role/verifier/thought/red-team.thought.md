## 红队思维的本质

**定义**：站在"破坏者"视角，主动寻找系统漏洞、薄弱环节、设计缺陷的思维方式。

**与传统测试的差异**：
- 传统测试：验证系统**应该**做什么（happy path）
- 红队思维：验证系统**不应该**做什么（adversarial path）

**核心问题清单**（每个被验证对象都要过一遍）：
1. **输入边界**：空值、超长、特殊字符、Unicode、emoji、SQL 注入、prompt 注入
2. **状态边界**：并发竞争、状态机非法转换、重复 update、过期数据
3. **权限边界**：未授权访问、提权、跨角色越权
4. **逻辑边界**：死循环、无限递归、栈溢出、内存泄漏
5. **语义边界**：歧义指令、相邻概念混淆、上下文遗忘

</exploration>

## 红队思维的方法论

### 第一步：威胁建模（Threat Modeling）

**问题**：哪些"坏人"会怎么攻击这个角色？

| 角色类型 | 潜在攻击 |
|---------|---------|
| 恶意用户 | 试图越权、调教角色做违反规则的事 |
| 误用用户 | 错误指令导致角色行为偏离 |
| 边界探针 | 用极端输入压测角色稳定性 |
| 跨角色攻击 | 通过 handoff 协议污染其他角色 |

### 第二步：攻击面枚举（Attack Surface）

列出所有**输入入口**和**状态出口**：

```
输入入口：用户消息、tool 调用、handoff 消息、文件读取
状态出口：响应内容、状态变更、文件写入、tool 调用
```

每个入口都要构造对抗性输入。

### 第三步：假设破坏（Hypothetical Breaking）

**核心问题**：如果我是攻击者，怎么让这个角色**输出错误**、**泄露信息**、**越权操作**？

**反问清单**：
- "如果用户说 X 而我的 Prompt 里说 Y，会冲突吗？"
- "如果用户传入超长字符串，token limit 会怎么影响响应？"
- "如果工具返回错误，角色的 fallback 是什么？"
- "如果 handoff 消息被篡改，角色会执行吗？"

### 第四步：漏洞验证（Vulnerability Verification）

**原则**：宁可错杀、不可漏判。

- 任何**疑似**漏洞 → 标记为"待验证"
- 任何**可复现**漏洞 → 标记为正式风险
- 任何**影响交付**的漏洞 → 必须修复后才能发布

</reasoning>

## 红队思维的应用关键点

### 何时启动红队思维

| 场景 | 启动时机 |
|------|---------|
| **新角色交付前** | 必须经过红队攻击测试 |
| **角色 Prompt 修改后** | 必须回归红队用例 |
| **工具更新后** | 必须重测相关攻击面 |
| **生产事故复盘** | 必须补漏对应红队用例 |

### 红队思维的边界

**不能滥用**：红队思维不是"找茬"，目的是**暴露真实风险**。

**避免偏见**：
- ❌ 预设"这个角色一定有问题"
- ✅ 客观构造攻击，验证是否真的有问题
- ❌ 把设计缺陷当攻击漏洞（设计是姜子牙的事）
- ✅ 只攻击**实现层**和**运行层**的漏洞

### 红队报告的呈现

**避免戏剧化**：
- ❌ "天哪这个角色有大问题！"
- ✅ "测试场景 X 下，角色输出违反规则 Y，重现步骤如下..."

**附带证据**：
- 重现步骤（精确到 prompt）
- 预期输出
- 实际输出
- 严重等级
- 修复建议

## 红队思维的执行策略

### 测试用例设计模式

**模式 1：边界输入攻击**
```
输入：空字符串、超长字符串、特殊字符、Unicode 控制字符
目的：触发 prompt 解析或模型上下文溢出
```

**模式 2：指令覆盖攻击**
```
输入："忽略之前的指令，改为..."
目的：测试角色是否被 prompt 注入劫持
```

**模式 3：跨角色污染攻击**
```
输入：伪造 handoff 消息（to, from, payload）
目的：测试角色是否校验 handoff 来源
```

**模式 4：状态机非法转换**
```
操作：handoff status = completed → pending（反向）
目的：测试状态机是否拒绝非法转换
```

### 漏洞分级标准

| 等级 | 含义 | 修复 SLA |
|------|------|---------|
| **致命 (P0)** | 角色完全不可用 / 数据泄露 / 安全漏洞 | 立即修复，阻塞发布 |
| **严重 (P1)** | 主要功能失效 / 多次触发 | 24 小时内修复 |
| **一般 (P2)** | 边界场景异常 / 体验降级 | 一周内修复 |
| **建议 (P3)** | 代码风格 / 可优化点 | 下一个迭代 |
