## 结构化测试的本质

**定义**：基于**测试用例库**和**规范模板**，系统化执行验证流程，确保覆盖率的思维方式。

**与红队思维的关系**：
- **结构化测试**：自上而下，**证明系统能做什么**（基于规范）
- **红队思维**：自下而上，**证明系统不能做什么**（基于攻击）

**两者互补**：
- 结构化测试覆盖率 = 高
- 红队思维漏洞率 = 低
- 双引擎 = 既验证 happy path，又覆盖 adversarial path

</exploration>

<reasoning>

## 结构化测试的方法论

### 第一步：需求拆解（Requirement Decomposition）

从角色设计文档中提取**可验证点**：

```
角色设计文档（姜子牙产出）
   ↓
五维模型分析
   ├─ 本质定位 → 验证角色定位描述与行为一致
   ├─ 能力体系 → 验证每个能力的触发和输出
   ├─ 认知模式 → 验证思维链路的完整性
   ├─ 价值系统 → 验证决策是否符合价值观
   └─ 表达风格 → 验证输出风格符合规范
   ↓
可验证点清单（每个点对应 1+ 个测试用例）
```

### 第二步：测试用例设计（Test Case Design）

**测试用例四要素**：
1. **输入**（input）：触发角色响应的 prompt 或操作
2. **预期**（expected）：角色应该输出的内容或行为
3. **实际**（actual）：角色实际输出的内容或行为
4. **判定**（verdict）：通过 / 失败 / 待定

**用例分类**：

| 类型 | 覆盖率目标 | 用例数量 |
|------|-----------|---------|
| **核心功能** | 100% | ≥ 每个能力 3 个 |
| **happy path** | 100% | ≥ 每个流程 2 个 |
| **异常路径** | ≥ 80% | ≥ 每个边界 1 个 |
| **回归基线** | 100% | 历史所有用例 |

### 第三步：用例库管理（Test Suite Management）

**测试用例库结构**（持久化于 `@!knowledge://test-case-patterns`）：
```
test-cases/
├── <role-id>/
│   ├── core-functionality.json    # 核心功能用例
│   ├── happy-path.json            # 正常路径用例
│   ├── error-path.json            # 异常路径用例
│   ├── regression-baseline.json   # 回归基线
│   └── last-run-<timestamp>.json  # 最近一次运行结果
```

**用例版本化**：
- 每次角色迭代前冻结当前用例库为 baseline
- 迭代后对比 baseline，发现回归

### 第四步：覆盖率分析（Coverage Analysis）

**覆盖率指标**：

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| **能力覆盖率** | 已验证能力数 / 总能力数 | ≥ 95% |
| **场景覆盖率** | 已验证场景数 / 总场景数 | ≥ 90% |
| **路径覆盖率** | 已验证执行路径数 / 总路径数 | ≥ 85% |
| **用例通过率** | 通过用例数 / 总用例数 | ≥ 98% |

**未覆盖分析**：任何指标 < 目标，必须补充用例或说明不可覆盖原因。

## 结构化测试的应用关键点

### 用例设计的"度"

**避免过度测试**：
- ❌ 为每个 prompt 变体写一个用例（无穷无尽）
- ✅ 按**等价类**和**边界值**设计代表性用例

**避免测试盲区**：
- ❌ 只测 happy path，忽略异常流
- ✅ 必须为每个能力设计**至少 1 个异常用例**

### 用例库的维护成本

**问题**：用例库会无限膨胀，怎么办？

**策略**：
1. **定期归档**：超过 3 个版本未触发的用例 → 归档
2. **合并等价用例**：相同触发条件、不同输入的用例 → 合并为参数化用例
3. **优先级排序**：核心用例长期保留，边缘用例定期清理

### 测试结果的可信度

**问题**：LLM 行为存在随机性，怎么保证测试结果可信？

**策略**：
1. **多次运行取众数**：同一用例运行 N 次（N≥3），取最频繁结果
2. **温度参数固定**：测试时 `temperature=0`，确保结果稳定
3. **关键用例人工复核**：核心功能用例必须有**人类 reviewer** 签字

## 结构化测试的执行策略

### 测试流程模板

```
1. 准备阶段
   - 加载角色定义
   - 加载测试用例库
   - 配置运行环境

2. 执行阶段
   - 按顺序执行测试用例
   - 记录每次输入/输出
   - 标记超时和异常

3. 分析阶段
   - 对比预期/实际
   - 计算覆盖率指标
   - 生成风险清单

4. 报告阶段
   - 输出结构化报告
   - 标注修复优先级
   - 反馈给实现方（女娲/鲁班）
```

### 测试用例模板

```json
{
  "case_id": "smoke-001",
  "category": "core-functionality",
  "priority": "P1",
  "input": {
    "role": "verifier",
    "prompt": "请验证角色 nuwa 的 DPML 合规性",
    "context": {}
  },
  "expected": {
    "contains": ["角色定位", "五维模型", "协作边界"],
    "not_contains": ["未实现", "TODO"],
    "structure": "structured"
  },
  "actual": null,
  "verdict": "pending",
  "last_run": null
}
```

```markdown
# 验证报告：<role-id> v<version>
```
